## 1. プロダクトの旅における Tokiq の現在地

```
   顧客 ▸ 課題 ▸ 価値 ▸ [ ★ 現在地：検証 ] ▸ PMF ▸ 成長 ▸ スケール
```

私たちには **アイデアと説得力のあるプロトタイプ** がありますが、**検証済みの需要はゼロ** です。
十分な数の実プレイヤーにまだ話を聞けておらず、プロトタイプの対戦相手はシミュレーションなので、
継続率や支払い意欲についてはまだ何も主張できません。**本計画は、その不確実性を安価に取り除く
ためのものです。**

> 本計画の *根拠*（プロダクトの考え方、5 つの学習フェーズ、PMF への 7 ステップ、「リサーチ
> ボード」の問い）は、**文書 1 の §2「Tokiq のようなプロダクトの考え方」** にあります。本書は、
> その思考を行動に変えます。

私たちが **まだ行わない** ことを明確にします。スケール、有料マーケティング、開発のための採用、
本物の決済連携、本格的なマルチプレイヤー基盤です。*フィットの前に拡大するのは「まだ着いていない
火にガソリンを注ぐ」ようなものです。*

## 2. 何を証明しようとしているのか（仮説）

以下のそれぞれを、前提ではなく **検証すべき主張** として扱います。各仮説には「検証済み／反証」の
明確なシグナルを定めます。

| # | 仮説 | 検証されたと言えるのは… |
|:---:|---|---|
| **H1 望ましさ** | 「目標に最も近く止める」ループは本当に面白く、緊張感がある。 | プレイヤーが促されずに「もう 1 回」を繰り返す。 |
| **H2 惜敗フック** | 負けても惜しいと、やめずに再挑戦したくなる。 | 惜敗の後、高い割合で再挑戦し、意欲をかき立てられたと語る。 |
| **H3 継続** | 負けたプレイヤーが別の日に戻ってくる。 | D1・D7 で意味のある割合が再訪する（§5 の目標参照）。 |
| **H4 支払い意欲** | プレイヤーは賞金プールのために参加費を払う。 | 支払いの意図を示し、ある参加費レンジを受け入れる。 |
| **H5 拡散・社会性** | プレイヤーは友人を誘い、勝つ姿を見られたがる。 | 機会を与えると結果を共有・紹介する。 |
| **H6 差別化** | プレイヤーは Aviator の運より Tokiq のスキル感を好む。 | Tokiq を選ぶ理由を言語化でき、直接比較で好む。 |
| **H7 信頼** | 「証明可能な公正性」は重要で、かつ信じられる。 | 信頼・公正が重要として挙がり、説明が理解される。 |
| **H8 スキル感** | プレイヤーは上達できると信じる。 | セッションを重ね、習熟・上達の手応えを語る。 |

> いま最も重要なのは **H1（面白いか？）と H3（戻ってくるか？）の 2 つ** です。これらが満たされ
> なければ、他は意味を持ちません。H4・H7・規制の論点は、実際の有料ローンチの前に重要となります。

## 3. 手法：最も安価な実験から

計画全体は 1 つのファネルです。最も安価で速いテストから始め、安価なシグナルが正当化して初めて
本物のエンジニアリングに投資します。各段階は意思決定ゲート（§6）につながります。

```
   安価＆高速
   (聞く・観る)
        │
        ▼
   ┌─────────────────────────────┐
   │ A · 顧客インタビュー          │
   │ B · プロトタイプの観察テスト   │ ──▶  ゲート1  面白いか？惜敗フックは効くか？
   └─────────────────────────────┘              (H1, H2, H8)
        │ 通過
        ▼
   ┌─────────────────────────────┐
   │ C · ランディングページ(需要)   │
   │ D · 支払い意欲テスト          │ ──▶  ゲート2  需要＋支払い意欲はあるか？
   └─────────────────────────────┘              (H4, H5, 需要)
        │ 通過
        ▼
   ┌─────────────────────────────┐
   │ E · 最小マルチプレイヤー試行    │ ──▶  ゲート3  実プレイヤーは戻り、誘うか？
   └─────────────────────────────┘              (H3, H5)
        │ 通過
        ▼
   難所をつくる (マルチプレイヤー、不正対策、決済、規制)
   高コスト＆低速、ただし今や正当化される
```

### 実験 A：顧客発見インタビュー *(最も安価、最初に行う)*
- **対象：** ターゲット層 10〜15 名、**現役の Aviator／クラッシュ系プレイヤー** 数名、後に **離脱
  ユーザー** 数名。離脱ユーザーから最も学べます。
- **方法：** **「The Mom Test」** の規律を用い、仮定ではなく **過去の行動** を尋ねます。
  - ✅ 「Aviator のようなゲームを最後に遊んだときの話を。何が起きましたか？」
  - ✅ 「先月この種のゲームにいくら使いましたか？ 具体的に何に？」
  - ✅ 「以前遊んでいた何かをやめた理由は？」
  - ❌ 「こういうゲームを遊ぶ／払うと思いますか？」（人は気を遣って嘘をつく）
- **検証：** H1, H4, H6, H7、および文書 1 §2.4 の 5 つのリサーチ質問。

### 実験 B：モデレート付きプロトタイプ観察テスト
- **対象：** 8〜12 名、1 人ずつ、観察しながら（対面または画面共有）。
- **方法：** 既存のプロトタイプ（https://slide-desk.vercel.app/）を **説明なしで** 渡します。
  誘導せず観察します。目的を理解するか？ 促されず「もう 1 回」をタップするか？ どこで笑い、うめき、
  前のめりになるか？ いつ飽きるか？
- **検証：** H1, H2, H8。
- **正直な限界：** 対戦相手はシミュレーションなので、これは **単独プレイの手触りと惜敗フック** を
  測るものであり、真のマルチプレイヤー継続率ではありません。

### 実験 C：ランディングページ／「フェイクドア」需要テスト
- **内容：** Tokiq を説明し「ウェイトリスト登録／先行アクセス」ボタンを置いた簡素なページ。任意で
  少額・1 チャネルのみの集客テストを行う。
- **計測：** 訪問→登録の転換率、どのメッセージが響くか。
- **検証：** 需要シグナル、および H5（リンクを共有するか？）。室外の誰かが関心を持つかを安価に見る。

### 実験 D：支払い意欲のスモークテスト
- **内容：** プロトタイプまたはランディング導線で、参加費の選択（まず仮想マネー、次に *疑似* 有料
  フロー）を複数の価格帯で提示し、離脱を観察する。
- **計測：** どの参加費でためらうか？ プールは参加に値するか？
- **検証：** H4。
- **ガードレール：** 規制チェック（§7）が完了するまで **実際の金銭を受け取らない。**

### 実験 E：最小マルチプレイヤー試行 *(A〜D が良好な場合のみ)*
- **内容：** 招待した少人数で、最小限の **本物の** マルチプレイヤーラウンド（粗くても可）を行い、
  *初めて* の真の継続・社会性シグナルを得る。
- **検証：** H3 と H5 を本当に検証。本物のエンジニアリングを要する最初の実験であり、だからこそ最後
  に、安価なシグナルが正当化してから行います。

## 4. Tokiq 固有の設計・観察ポイント

- **アハ体験**（プレイヤーが「わかった」と価値を感じる瞬間）。確認すべき候補：初の惜しいストップ、
  初勝利、初のリーダーボード入り、初の社会的承認。*観察テストで本物を見つけ、そこへ素早く到達でき
  るよう設計する。*
- **フックループ：** *トリガー → 行動 → 変動報酬 → 投資。* 変動報酬は結果の「近さ」の予測不能性
  です（図は文書 1 §5）。
- **「敗北を許容可能に、勝利を記憶に残す」。** 敗北の感触を観察します。惜敗は再挑戦を誘うべきで、
  離脱を招いてはいけません。

## 5. 重要な指標（と無視すべき指標）

**これらを追う（アウトカム）：**

| 指標 | なぜ重要か | 初期の目安* |
|---|---|---|
| **「もう 1 回」率** | 中核ループの引力 | 多くのプレイヤーが促されず 3 回以上遊ぶ |
| **D1 / D7 継続率** | 成否を分けるシグナル | まず基準値を作り、その後改善する |
| **セッションあたりラウンド数／滞在時間** | エンゲージメントの深さ | 来訪を重ねて上昇傾向 |
| **Sean Ellis スコア** | PMF の代理指標：「Tokiq が使えなくなったらどう感じるか？」 | **40% 以上「非常に残念」** を目指す |
| **紹介・共有率** | 組み込み型成長の可能性 | 自然な共有はどれも強いシグナル |
| **惜敗→再挑戦率** | フック（H2）を検証 | 僅差の敗北後に高い再挑戦 |

\* *目標はチームで磨くための出発点であり、確約ではありません。*

**これらは無視する（虚栄の指標）：** ダウンロード数、ページ閲覧数、インストール数。

```
   ✗ 100 万ダウンロード、誰も戻らない   ──▶  死んでいる
   ✓        100 人、毎日遊ぶ           ──▶  生きている
```

## 6. 意思決定ゲート（Go / No-Go）

検証は一発勝負ではなく、一連のチェックポイントです。各ゲートで **継続・方向転換・中止** の
いずれかを決めます。

```
   ゲート1   A + B の後   ─►  ループは面白いか？惜敗フックは効くか？  (H1, H2, H8)
            │  否  ──▶ 中核メカニクスを方向転換、または中止
            └─ 可  ──▶ 継続

   ゲート2   C + D の後   ─►  需要と支払い意欲はあるか？           (H4, H5)
            │  否  ──▶ ポジショニングまたは対象を再考
            └─ 可  ──▶ 継続

   ゲート3   E の後       ─►  実プレイヤーは戻り、他者を誘うか？     (H3, H5)
            │  可  ──▶ 今こそ難所に投資し、事業・GTM 計画を書く
            └─ 否  ──▶ さらに投資する前に継続設計を見直す
```

> 本格的なエンジニアリングの費用を投じるのは **ゲート3 の後だけ** です。それ以前はすべて安価です。

## 7. 並行して進める：規制と公正性のトラック

Tokiq は参加費と賞金プールを伴うため、2 つのワークストリームを検証と **並行して**（後ではなく）
進める必要があります。いずれもモデル全体を変えうるからです。

1. **法務・規制レビュー。** 市場ごと（**特に日本**）と、対象とする他市場（例：東アフリカ）における
   スキル対賭博の分類。スキルベースの位置づけは助けになりますが、適用除外を **保証はしません**。
   *担当者と、必要に応じて外部顧問。*
2. **公正性の設計。** 信頼できる **「証明可能な公正性」** の方式（例：Aviator のようにプレイヤー
   シードを用いた SHA-256 のコミット・リビール）と、端末側の不正を防ぐための **サーバー側の権威
   ある計測 ＋ 遅延処理** の計画。これは中核の技術課題であり、検証期間中は *スコープ化*（構築は
   しない）すべきです。

## 10. チーム議論で決めたいこと

1. これが **正しい段階**（つくるのではなく検証する）であり、2 つの文書でカバーできていると合意する。
2. **仮説**（§2）を批判的に検討する。欠けや誤りはないか？
3. **指標の目標**（§5）と **ゲート**（§6）を点検する。
4. **担当**（§9）を割り当て、**スケジュール**（§8）を確定する。
5. **規制の初読**（§7）を誰が始めるかを決める。これはモデルを静かに葬る／作り替えうるため、早期に
   着手したい。

> **結論。** 最大のリスクは Tokiq を *つくれるか* ではありません。プレイヤーが *戻り、信頼し、お金
> を使い、共有するか* です。本計画は、次の数週間をかけてそれをできるだけ安価に明らかにし、次の大きな
> 意思決定を証拠に基づいて行えるようにします。
